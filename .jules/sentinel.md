## 2025-02-26 - Prevent XSS from URL parameters via DOM insertion
**Vulnerability:** XSS vulnerability in `SocialMediaPage.tsx` caused by directly interpolating URL parameters (`result.username`, `result.message`) into a template string passed to `.innerHTML` for toast notifications.
**Learning:** URL parameters fetched during OAuth callbacks can contain malicious payloads. If directly assigned to `.innerHTML`, script tags or other malicious HTML can be executed by the browser. Even simple fields like `username` or `message` should be considered untrusted.
**Prevention:** Never use template literals containing dynamic values with `.innerHTML`. Always create empty containers (e.g., `<span class="message-container"></span>`) and then use `.textContent` or `.innerText` to securely inject dynamic content, ensuring it is treated as text, not executable HTML.

## 2025-03-09 - Add payload signature validation to external webhook endpoints
**Vulnerability:** Meta webhook endpoints in `api/webhooks.ts` lacked payload signature validation (using `X-Hub-Signature-256`) and defaulted to an insecure hardcoded verify token if unset. This allowed arbitrary payload injection and bypassed Meta's webhook security model.
**Learning:** Webhook endpoints must mathematically verify that incoming payloads originated from the expected service. For Meta webhooks, this requires computing an HMAC SHA-256 signature using the raw request body and the `INSTAGRAM_CLIENT_SECRET`. Express's `express.json` middleware strips the raw buffer unless explicitly captured using its `verify` function.
**Prevention:** Never use hardcoded fallback tokens for webhook verification. Always capture the raw request buffer in Express using `express.json({ verify: (req, res, buf) => { req.rawBody = buf; } })` to ensure accurate signature computation. Always validate incoming webhook signatures using `crypto.timingSafeEqual` to prevent timing attacks.
