## Architecture

The pipeline uses a **modular subagent architecture** — a sonnet dispatcher spawns focused haiku phase subagents using their **registered `subagent_type`** (which auto-loads each agent file). No Opus tokens are consumed.

```
Command (/task-processor)
  -> Dispatcher (task-processor.md, SONNET) — orchestrates phase transitions
      -> Phase 1: task-processor-discover (haiku) — MCP calls only
      -> Phase 2: task-processor-analyze (haiku) — writes prompt, fires Gemini clink
      -> Phase 3: task-processor-plan (haiku) — writes prompt, fires Gemini clink
      -> Phase 4: task-processor-decompose (haiku) — spawns task-decomposer per task
      -> Phase 5: task-processor-execute (haiku) — fires concurrent Gemini clinks per wave
      -> Phase 6: task-processor-review (haiku) — reads files, checks severity checklist
      -> Phase 7: task-processor-fix (haiku) — surgical Gemini clinks for CRITICAL/HIGH
      -> Phase 8: task-processor-complete (haiku) — MCP calls, cleanup
```

Each phase is spawned with its matching `subagent_type` (e.g., `"task-processor-analyze"`), which auto-loads the agent file. The dispatcher also inlines critical rules (clink conventions, project root, state file path) in every spawn prompt.

Shared context (project stack, rules, state schema) lives in `task-processor-shared.md` — each subagent reads it as its mandatory first step.

## Workflow

Use the **task-processor** agent (`.claude/agents/task-processor.md`) to execute the full 8-phase pipeline:

1. **DISCOVER** — Recover stale in-progress tasks, fetch 3-6 pending tasks, filter by satisfied dependencies, mark in-progress
2. **ANALYZE** — Gemini analyzes codebase via clink MCP Call (write prompt-analyze.txt, fire clink)
3. **PLAN** — Gemini creates implementation plan via clink MCP Call (write prompt-plan.txt, fire clink)
4. **DECOMPOSE** — Spawn `task-decomposer` haiku subagent per task to produce wave-based micro-task breakdowns
5. **EXECUTE** — Process waves: within a wave, fire ALL micro-clinks CONCURRENTLY; across waves, sequential
6. **REVIEW** — Haiku subagent reads modified files and reviews against severity checklist (NO clink)
7. **FIX** — Surgical micro clink MCP calls for CRITICAL/HIGH issues only (max 2 cycles)
8. **COMPLETE** — Mark tasks done in Task Master, log implementation notes, delete state file

## Resume Mode

If a previous session was interrupted, `/task-processor` automatically resumes:

- Reads `.taskmaster/processor-state.json`
- Skips completed phases, waves, and micro-tasks
- Continues from the last checkpoint
- No arguments needed — state file controls everything

To **force a fresh start** (discarding checkpoint): `/task-processor reset`

## Arguments

- No arguments: run full pipeline on next available batch (or resume if checkpoint exists)
- `reset`: delete `.taskmaster/processor-state.json` and start fresh
- `dry-run`: run Phases 1-4 (DISCOVER, ANALYZE, PLAN, DECOMPOSE) only — show wave breakdown without executing
- `status`: show current checkpoint state without running anything
- Any other text: passed as context filter (e.g., task IDs to focus on, or "backend-only" / "frontend-only" hints)

## Abort

If a clink call is stuck or running too long:

1. Press Ctrl+C to interrupt the current tool call
2. The state file preserves all progress — nothing is lost
3. Run `/task-processor` again to resume from checkpoint

Do NOT try to kill processes via PowerShell/cmd from Git Bash — it doesn't work. Use Task Manager or a native terminal instead.
